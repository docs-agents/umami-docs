---
title: 从 v1 迁移到 v2
---

Umami v2 引入了重新设计的架构和一些重大变更。

## 重大变更

- API 端点已更改，包括新增的端点。有关更多信息，请参阅 [API](/docs/api)。
- 追踪脚本已从 `umami.js` 重命名为 `script.js`。
- 收集 API 端点已从 `/api/collect` 重命名为 `/api/send`。
- 追踪器不再使用 CSS 类名来触发事件。它现在使用 HTML 数据属性。有关更多信息，请参阅[追踪事件](/docs/track-events)。
- 全局 `umami` JavaScript 对象上的方法已更改。现在有一个单一的 `.track()` 方法和一种发送事件数据的新方式。
  有关更多信息，请参阅[追踪器函数](/docs/tracker-functions)。
- `TRACKER_SCRIPT_NAME` 环境变量不再追加 `.js` 扩展名到脚本名称。请参阅[环境变量](/docs/environment-variables)获取更多信息。

## 数据迁移

由于架构变更，您 v1 数据库中的数据需要转换为 v2。
为协助迁移，我们创建了一个脚本 `@umami/migrate-v1-v2`，它将为您迁移所有数据。

### 要求

- 数据库架构必须与最新 v1 版本（v1.4.0）同步。该脚本将查询 Prisma 迁移表以确保已运行最新迁移。

### 重要

- 使用前备份目标数据库。如果迁移被中断，可能会发生潜在的数据丢失。
- 对于数据集较大（500 万以上）的用户，迁移可能需要一段时间。请相应地进行计划。
- 该脚本不会将任何事件数据迁移到 v2。
- 该脚本会询问您是否要在迁移完成后删除 v1 表。
- 如果找到 `event_data` 表填充了数据，它将被重命名为 `v1_event_data` 但不会被删除。

### 故障排除

- 如果您的 `DATABASE_URL` 是 localhost 且迁移无法连接到数据库，请尝试更改为 IP 地址，例如：`127.0.0.1`。

## 运行迁移

有两种方式运行迁移脚本。

### 在 Umami 文件夹内运行

如果您可以访问应用程序文件夹，请使用此方法。
确保您的应用程序已经构建。如果没有，请先运行 `yarn build`。

```shell
cd umami
npx @umami/migrate-v1-v2@latest
```

### 独立运行

如果您无法访问应用程序文件夹（如部署到 Vercel 或 Netlify 时），请使用此方法。

**安装**

```shell
git clone https://github.com/umami-software/migrate-v1-v2.git
cd migrate-v1-v2
yarn install
yarn build
```

**配置**

创建包含以下变量的 `.env` 文件：

```
DATABASE_URL={连接 URL}
```

**运行**

```shell
yarn start
```

## Docker 迁移

进入您的运行中的 Docker 容器。您可以通过 `docker ps` 的输出找到名称。

```shell
docker exec -ti -u 0 <app container name> sh
```

运行迁移脚本。

```shell
npx @umami/migrate-v1-v2@latest
```
